Project Goals and Description

The intention behind the build for Spa Selene was to apply front end patterns I had been learning in JavaScript and implement them together to form a full site. This process allowed me to gain greater confidence problem solving with JavaScript and continue flexing my CSS layosut muscles. Due to the nature of the buisness the website was chosen to reflex, there were some essential features I knew had to be incorported. First, the website needed to feature a menu component and display related content filtered by category. Also, a form validation component was necessary to facilitate the exchange of customer information to reserve a service. In my mind, these two features formed the primary development goals for the site with more general components like an about section, a frequently asked questions section, and a reviews section emplyed to add legitamecy and well rounded aspects to better inform the site's purpose to users.

Development Process

For any project I build from scratch, I start by creating quick sketches of my vision for the UI and annotate it with my ideas for the functionality. This helps the design process go much faster and smoother as I can easily throw out a 1-minute sketch and draft another as opposed to wasting time coding something up I then decide isn't appealing. This also really helps me for planning purposes so I can get a big picture of the application and then move forward in identifing things I know I can tackle quickly as opposed to things I want to accomplish but don't have as much experience implementing. After the planning stage, I coded up my HTML markup with the bones of the design and identified color and font themes to start constructing my SASS variables. As I progressed through the build of the layout, I primarily utilized CSS grid.


In the project description, I mentioned there were a number of javascript patterns I knew would be compatible components for this application. I was confident in a number of the these elements due to prior study and mini-projects I've worked through and completed. These features namely being:


  • the responsive and fixed navigation
  • the accordion component
  • the reviews component
  • and process of displaying a category of data in response to a click event

As a result, I was able to build all of the above features for the site in four days. It would have happened much faster but I wanted to explore how to make the UI for these items more interesting. This involved my time spent learning how to effectively manipulate background images, animate items in on scroll, and add a mouse hover effect from behind the menu cards. The learning process I followed for all unknown functionality I wanted to implement in the project involved a combination of reading documentation on Mozilla Developer Network, reviewing existing code (via StackOverflow, CodePen, Github), reading tutorials on CSS tricks, and watching youtube video tutorials. Once I was confident in how the code worked, I would build the feature in a seperate standalone file to make sure I understood what I learned and could trouble shoot more easily. Then, I would build the new feature into the main project.


Two specific patterns I knew needed to be implemented but was less clear on how to accomplish was the multistep form validation and its integration with the navigation bar to generate a popup in response to a click. I had completed single step form validation multiple times at this point, so I was excited for the challenge. I watched youtube tutorials and reviewed several codepen's in preparation. I came to find the difficult part was finding a build that was written in vanilla javascript that applied unique validation functions for input and select elements. I ended up compiling different aspects from this learning phase combined with refactoring sessions to achieve the validation I wanted for this multistep form functionality.

Feature Analysis - Booking Component

My favorite aspect of this project, and the most challenging, was the booking component. The steps to accomplishing the functionality I had in mind for the form are as follows:


  • only certain elements are required
  • only certain options will be displayed in response to a given option being selected
  • avoid the automatic HTML form validation in favor of custom functions
  • those custom functions will communicate and style elements for an error or success
  • only run custom functions on the elements related to a corresponding form step
  • determine if all of the function validations returned a success
  • if all successful, let a button click signal the next step in the form
  • if any unsuccessful, stay on current step and display error messages

The first two goals were accomplished through a simple class of required and switch statement. Custom validation functions were then created in response to associated required elements. The validation functions test the elements value and pass it to an error/success function for styling.


Its critial for the functions tied to certain inputs to only run during the matching step of the form. To do this I created a function that would apply a class of "active" to a given form step/phase based on its order in the HTML.


Then I was able to use a nested if statement to ensure if an element is required and it's on a certain form step, to run specific custom functions. If the result of those function calls gave us an array where all elements had a class of success, then a next button click would increment the current form step and assign the class of active to it.


After my work on this component, I have an improved understanding on form validation and can better ensure only proper information will be shared with a server.
